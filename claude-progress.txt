## Détail Session 3 (complété le 21 mars 2026)

### Packages installés
- `ai` (Vercel AI SDK) + `@ai-sdk/openai` (provider OpenAI-compat pour OpenRouter)

### Migration — fix RLS Clerk JWT
**supabase/migrations/20260321000001_fix_rls_clerk_jwt.sql**

Problème résolu : auth.uid() retourne NULL avec les JWTs Clerk (clerk_id = "user_xxx", pas un UUID).
Solution : policies réécrites avec `(auth.jwt() ->> 'sub')` (comparaison TEXT via clerk_id).

Tables mises à jour : users, projects, service_connections, conversations, messages, credit_transactions.

Nouvelle fonction RPC Postgres : `deduct_credits(p_user_id, p_amount, p_description, p_project_id?)`
- SECURITY DEFINER (droits service role)
- Atomique : lock FOR UPDATE + UPDATE users + INSERT credit_transactions en une transaction
- Lance exception `insufficient_credits:` si solde insuffisant
- REVOKE ALL FROM PUBLIC (seul le service role peut appeler)

**PREREQUISITE** : appliquer cette migration + configurer Supabase pour accepter les JWTs Clerk :
- Supabase Dashboard → Project Settings → Auth → Third-party auth providers → Add Clerk
- JWKS URL : https://<clerk-domain>/.well-known/jwks.json

### src/lib/supabase.ts — mise à jour

```

  ├── Stage 3: Supabase schema
  │     REST API pg-meta + rpc fallback → split SQL statements → exec séquentiel
  │
  └── Stage 4: Persist
        supabase.projects.update(status="deployed", stack_config={urls})
```

### Règles de sécurité respectées
- Tokens déchiffrés en mémoire au moment de l'usage, jamais persistés en clair
- sanitizeForLog sur TOUS les messages de log et d'erreur
- Env vars client injectées sur LEUR Vercel, pas sur le nôtre
- L'agent ne voit JAMAIS les tokens (il reçoit seulement "tu as accès au GitHub du client")
- Erreur partielle : si Vercel échoue, le code GitHub est conservé (ownership client)
